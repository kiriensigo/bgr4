# マスターキーをGoogle Cloud Secretsマネージャーに保存するコマンド

## 1. シークレットマネージャーが有効化されていることを確認

```powershell
# Secret Manager APIを有効化
gcloud services enable secretmanager.googleapis.com
```

## 2. シークレットの作成と値の保存

```powershell
# シークレットを作成
gcloud secrets create rails-master-key --replication-policy="automatic"

# シークレットの値を追加（マスターキーを設定）
echo "cf8dee1c404263fe64df934d04335c9f" | gcloud secrets versions add rails-master-key --data-file=-
```

## 3. シークレットが正しく保存されたか確認

```powershell
# シークレットのリストを表示
gcloud secrets list

# シークレットの最新バージョンを取得（確認用）
gcloud secrets versions access latest --secret=rails-master-key
```

## 4. Cloud Runサービスにシークレットへのアクセス権を付与

```powershell
# サービスアカウントを特定
PROJECT_ID=$(gcloud config get-value project)
SERVICE_ACCOUNT=$(gcloud run services describe bgr4-api-minimal-rails --platform managed --region asia-northeast1 --format="value(serviceAccountEmail)")

# シークレットアクセサーのロールを付与
gcloud secrets add-iam-policy-binding rails-master-key \
  --member="serviceAccount:$SERVICE_ACCOUNT" \
  --role="roles/secretmanager.secretAccessor"
```

## 5. CloudBuildサービスアカウントにシークレットへのアクセス権を付与

```powershell
# CloudBuildサービスアカウントを特定
PROJECT_ID=$(gcloud config get-value project)
CB_SERVICE_ACCOUNT="${PROJECT_ID}@cloudbuild.gserviceaccount.com"

# シークレットアクセサーのロールを付与
gcloud secrets add-iam-policy-binding rails-master-key \
  --member="serviceAccount:$CB_SERVICE_ACCOUNT" \
  --role="roles/secretmanager.secretAccessor"
```

## 6. シークレットをローテーションする場合（定期的なセキュリティ対策）

```powershell
# 新しいバージョンのシークレットを追加（キーを変更する場合）
echo "新しいマスターキー" | gcloud secrets versions add rails-master-key --data-file=-

# 新しいバージョンのシークレットを使用するようにCloudBuildの設定を更新する
# （cloudbuild.yamlファイルのバージョン指定は不要。latestを使用していれば自動的に最新版を使用）
```

## 注意事項

- マスターキーを変更する場合は、credentials.yml.encファイルも再生成する必要があります
- 本番環境でキーを変更する場合は、移行計画を立てて慎重に行ってください
- Cloud Run以外のサービスを使用している場合は、それぞれのサービスにもアクセス権を付与してください
